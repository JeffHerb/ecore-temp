Deprecation Notes

Custom Tags

ET:List and ET:Combo

We have combinded the functionality of these to tags into <E2:OptionList>.

ET:JSButton

Developers should instead use the nearly-identical <E2:Button> custom tag and leverage its eventScript and preSubmitScript attributes as needed.

ET:DynaTabNavButton and ET:DynaTabNavList

There are no pages using this functionality at this time and the custom tag will not be replicated at this time.

Primary Button

The main difference between this button and a regular button was its inherit look and feel. This button has been deprecated adn the primary button styles can now be achived by using a normal <E2:Button> with the

ET:Output

It did not appear to be used in Empire 1 and no documentation could be found.

ET:TabNavPopup

The same functionality is now covered by <E2:TabNavButton> or <E2:TabNavLink>.

Attributes

ButtonTagId, pageElementName, notifierId

It appears that many people are using the same term viewKey for all of the listed above. For consistency all custom tags now use viewKey instead.

validate

Starting with Empire 2 all validation related to a field is controlled via the custom tag's viewKey. This means that only fields that have been properly registered via the Framework Registration Tool will receive validation. The JSON will contain an additional data-validation property in the attributes object, whose value is a comma-separated string of function names; and an additional className of cui-validation-field. See example below for more details: 

{
    "type": "text",
    "template": "field",
    "label": {
        "text": "Validatin Field",
        "attributes": {
            "for": "validationDemo"
        }
    },
    "input": {
        "attributes": {
            "name": "validationDemo",
            "id": "validationDemo",
            "class": "cui-validation-field",
            "type": "text",
            "value": "",
            "data-validation": "test1,test2,test3"
        }
    }
}
